java概述

JDK

​ 即Java Development Kit，是Java开发工具包，其中包含了JRE和一些开发中可能用到的工具（比如Java的编译器Javac等）。

JRE

​ 即Java Runtime Environment，是Java的运行环境，其内部包含JVM和一些类库。

JVM

​ 即Java Virtual Machine，是Java虚拟机，Java程序运行在这个上面，引入它使得Java程序跨平台时无需修改和重新编译即可运行，后面会有对JVM的详解。

Java的异常分类

  • 今天是2022年的9月13号啊，今天我刚明白过来java的异常分类到底是个什么意思，实在是惭愧惭愧。
  • 众所周知，java的异常分为非受检异常（运行时异常）和受检异常（编译时异常）。
  • 非受检异常有Error及其子类、RuntimeException及其子类，剩下的都是受检异常。
  • 这个受检异常是个什么意思呢，就是说你这个异常，在编译阶段会被IDE亲切的提醒你要显式地处理它，因为Java觉得你这个地方可能会出现异常，这也是它被称为编译时异常的原因，因为如果你不显式处理他，你的代码就过不了编译。
  • 再看非受检异常，它一般是因为你的代码逻辑出了问题，从而导致了java代码在运行时出现了问题，这个问题被jvm发现了，所以jvm要抛出异常提醒你这个地方出问题了，而这种异常一般是不会在编译阶段被Java发现的，毕竟Java的分析能力是有限的，总不能先预运行一下找找异常存不存在吧，那样效率也太低了。
  • 直到今天我才分清这俩的区别，之前我一直以为所有被抛出并打印到终端上的都是受检异常而非受检异常就是运行时遇到了程序直接寄了啥也不打印的那种，现在想想这个想法也太逆天了。

Java流式编程

  • 这是Java8中新支持的编程范式，我觉得很优雅，所以记录一下。
  • 流式操作，是 Java 8除了Lambda表达式外的又一重大改变。学习流式操作，就是学习java.util.stream包下的API，我们称之为Stream API，它把真正的函数式编程引入到了 Java 中。
  • 本小节我们将了解到什么是Stream，为什么使用Stream API， 流式操作的执行流程，如何实例化Stream，Stream的中间操作、Stream的终止操作等内容。
  • 基本上是用这个Stream操作来代替对集合的迭代操作的，但是效率比for循环稍低。

什么是Stream

  • Stream是数据渠道，用于操作数据源所生成的元素序列，它可以实现对集合（Collection）的复杂操作，例如查找、替换、过滤和映射数据等操作。
  • 我们这里说的Stream不同于java的输入输出流。另外，Collection 是一种静态的数据结构，存储在内存中，而Stream是用于计算的，通过CPU实现计算。注意不要混淆。
  • Stream自己不会存储数据；Stream不会改变源对象，而是返回一个新的持有结果的Stream（不可变性）；Stream操作是延迟执行的（这一点将在后面介绍）。

为什么使用Stream API

  • 我们在实际开发中，项目中的很多数据都来源于关系型数据库（例如 MySQL、Oracle 数据库），我们使用SQL的条件语句就可以实现对数据的筛选、过滤等等操作；
  • 但也有很多数据来源于非关系型数据库（Redis、MongoDB等），想要处理这些数据，往往需要在 Java 层面去处理。
  • 使用Stream API对集合中的数据进行操作，就类似于 SQL 执行的数据库查询。也可以使用Stream API来执行并行操作。简单来说，**Stream API提供了一种高效且易于使用的处理数据的方式**。

流式操作的执行流程

  • 流式操作通常分为以下3个步骤：

    1. 创建Stream对象：通过一个数据源（例如集合、数组），获取一个流；
    2. 中间操作：一个中间的链式操作，对数据源的数据进行处理（例如过滤、排序等），直到执行终止操作才执行；
    3. 终止操作：一旦执行终止操作，就执行中间的链式操作，并产生结果。

Java关于i++是否是原子操作的讨论

  • 什么是原子操作，就是一个操作不能再分了，已经是最小单位了，这样的操作在多线程中会表现出绝对的线程安全。
  • 那么i++是不是个原子操作呢，在java中很明显它不是一个原子操作，因为首先java是支持多线程的，这就不得不考虑这个代码在多线程环境下的表现，再看操作本身，被java解析成的汇编操作是先取i的值，然后再加1，然后再把新值赋值回去，其本身是可以再被细分的，而且这种性质使得他在多线程的环境下是线程不安全的，所以综上所述，i++在java中并不是一个原子操作。

Java8新特性之方法引用

  • 可以把方法引用看作lambda表达式的简写形式，感觉就是一种语法糖，而且还是语法糖中的语法糖，因为lambda表达式本身就是一种语法糖，jvm根本不认识lambda表达式。
  • 一共就四种情况的方法引用，对应着四种lambda表达式的使用。

lambda表达式调用类的静态方法的情况

  • lambda：(args) -> Class.staticMethod(args)
  • method reference：Class::staticMethod

lambda表达式调用传入的实例参数的方法的情况

  • lambda：(obj, args) -> obj.instanceMethod(args)
  • method reference：ObjectType::instanceMethod

lambda表达式调用已经存在的实例的方法的情况

  • lambda：(args) -> obj.instanceMethod(args)
  • method reference：obj::instanceMethod

lambda表达式调用类的构造函数的情况

  • lambda：(args) -> new ClassName(args)
  • method reference：ClassName::new

总结

  • 通过上面这四种情况，基本能看出来，除了第二种情况，其余的情况lambda表达式的参数都是用在被引用的方法的参数上的，第二种情况也差不多，只不过多了第一个参数用来调用方法。这样记好记一点。

Java引用在JVM中是如何存储的？

  • 今天在看多态的时候，突然意识到一个问题，那就是Java的引用在JVM中是以什么样的结构进行存储的？如果说Java引用只是一个指向对象的地址的话，JVM按理说应该不知道引用对应类型的结构，那多态性是怎么体现出来的呢？所以我感觉Java引用应该是以某种特定的结构存储在JVM中的，但是我不知道我这个想法对不对。
  • 或者换句话说，如果说引用类型只是存放对象地址的话，那为什么还要区分不同类的引用类型？直接定义一种引用类型表示地址，之后所有对象的地址都通过声明这种类型的变量来进行存储不就行了，为什么还要区分不同类的引用呢？所以我感觉Java引用肯定是以某种特定的结构存储在JVM中的，我真的好疑惑啊。
  • 好像和虚方法有关，也就是说在使用多态的时候，两边类的方法入口都在JVM中存着，编译时编译器帮我们通过左边的引用类型的元数据定位到虚方法，运行时JVM帮我们通过右边的实际类型的元数据定位到方法，这也能解释为什么不同类型的引用可以获取其中的信息了，是编译器帮我们通过引用类型这个元数据找到那个类的所有信息。
  • 大体意思就是，引用确实只存了地址，能通过引用找到对应的方法是要归功于编译器通过引用类型这个元数据定位到JVM中这个类所在的区域来进行查找对应方法的。

调用wait()、notify()、notifyAll()方法需要当前线程持有当前对象的锁

  • 这是为了防止出现死循环调用。
  • wait()之后当前线程马上放开锁，并将该线程挂到当前对象的等待队列中。
  • notify()之后，会唤醒一个在当前对象等待队列中的一个线程，这个唤醒是不确定的，优先级越高越有可能先被唤醒。
  • notifyAll()之后，会唤醒所有在当前对象等待队列中的线程。

sleep()和wait()的异同

相同点

  1. 一旦执行，都可以使得当前线程进入阻塞状态。

不同点

  1. 两个方法声明的位置不同：sleep()在Thread类中声明，wait()在Object类中声明
  2. 调用的要求不同：sleep()可以在任何需要的场景下使用，wait()必须在拥有当前对象锁的同步代码块或同步方法中使用。
  3. 关于是否释放同步监视器（锁）：如果两个方法都使用在拥有当前对象锁的同步代码块或同步方法中，sleep()不释放同步监视器，wait()会释放同步监视器。
